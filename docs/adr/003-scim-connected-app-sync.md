# ADR 003: SCIM 2.0 identity sync between SaaS scaffolding and connected apps

**Status:** Accepted  
**Deciders:** Platform team  
**Date:** 2026-07-18

## Context

SaaS scaffolding is the source of truth for tenants, users, and role bindings. Riogentix is the first connected app that consumes this identity data. Today the sync is point-to-point: `apps/workers/src/provisioning/kubernetes-driver.ts` calls a Riogentix-specific internal endpoint (`/api/v1/internal/saas/tenant/{id}/role-assignments`) using a shared secret. This is correct for v1 but will not scale once we add a second connected app.

We need a generic, standard, and incrementally adoptable design.

## Decision

Use **SCIM 2.0** as the wire protocol between SaaS scaffolding and every connected app.

- **SaaS scaffolding** acts as a SCIM client / identity source.
- **Each connected app** exposes a SCIM server endpoint.
- **Users** are modeled as `SCIM User` resources.
- **Roles** are modeled as `SCIM Group` resources. A Group's `externalId` is the SaaS `Role.id`, and its `members` are the app-local user IDs that correspond to SaaS users. Permission codes and the system-role flag travel in a custom SCIM extension (`urn:saas-platform:params:scim:schemas:extension:role:2.0`).
- **SaaS scaffolding** keeps a generic `ConnectedApp` registry + a transactional outbox. Adding app #2 is a registry row + a SCIM server on the app side — no per-app code in SaaS.

This is the same pattern used by Okta, Entra ID, and Google Workspace to provision identity into heterogeneous apps.

## Consequences

### Good

- Decoupled: SaaS no longer knows Riogentix internals.
- Standard: onboarding future apps is "implement SCIM subset + validate JWT".
- Drift-healing: the sync is declarative full-state, so replays and reconcilers converge safely.
- Real-time propagation: role changes after provisioning emit outbox events and are pushed immediately.

### Bad

- Slightly more complex than a custom JSON payload.
- SCIM is not natively multi-tenant; we need a tenant-scoped convention (see below).
- Apps must be able to resolve users by `userName` (email) and store the `externalId` mapping.

## Incremental migration plan

1. **ADR** (this doc) — agree on contract.
2. **SaaS scaffolding** — generic SCIM client, outbox, and worker already exist; expose them by creating a `ConnectedAppInstance` for Riogentix and emitting outbox events on identity mutations. Keep the old `kubernetes-driver.ts` sync running behind a flag until cutover is verified.
3. **Riogentix** — add a SCIM 2.0 server subset under `/api/v1/scim/v2/tenants/{tenant_id}/...` that maps to the existing `SaasRole` / `SaasRoleAssignment` tables and reuses the existing user resolution / provisioning logic. Auth: `Authorization: Bearer <tenant-scoped token>`.
4. **Cutover** — flip the SaaS flag so Riogentix is synced via SCIM instead of the old internal endpoint. Run e2e tests against the acme tenant.
5. **Cleanup** — remove the old point-to-point endpoint and shared-secret auth once the new path is stable.
6. **Future** — add a periodic reconciler and, if desired, replace the shared token with per-app OAuth2 client-credentials or mTLS.

## SCIM contract details

### Base URL

```
/api/v1/scim/v2/tenants/{tenant_id}
```

All SCIM resources are tenant-scoped. The tenant ID is a UUID that comes from the SaaS control plane.

### Auth

`Authorization: Bearer <token>`  
The token is generated by the SaaS control plane, stored in the `ConnectedAppInstance` table, and passed to the app during provisioning (initially via the same secret channel already used for `RIOGENTIX_SAAS_INTERNAL_SECRET`).

### User resource

```json
{
  "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
  "id": "app-generated-uuid",
  "externalId": "saas-user-id",
  "userName": "alice@example.com",
  "emails": [{ "value": "alice@example.com", "primary": true, "type": "work" }],
  "active": true,
  "meta": {
    "resourceType": "User",
    "created": "2026-01-01T00:00:00Z",
    "lastModified": "2026-01-01T00:00:00Z",
    "location": "/api/v1/scim/v2/tenants/{tenant_id}/Users/{id}"
  }
}
```

- `userName` is the email and is the cross-system identity key.
- `externalId` is the SaaS `User.id`.

### Group resource

```json
{
  "schemas": [
    "urn:ietf:params:scim:schemas:core:2.0:Group",
    "urn:saas-platform:params:scim:schemas:extension:role:2.0"
  ],
  "id": "app-generated-uuid",
  "externalId": "saas-role-id",
  "displayName": "tenant_admin",
  "members": [{ "value": "app-user-id" }],
  "urn:saas-platform:params:scim:schemas:extension:role:2.0": {
    "permissions": ["users:update", "notes:read"],
    "isSystem": true
  },
  "meta": {
    "resourceType": "Group",
    "created": "2026-01-01T00:00:00Z",
    "lastModified": "2026-01-01T00:00:00Z",
    "location": "/api/v1/scim/v2/tenants/{tenant_id}/Groups/{id}"
  }
}
```

- `externalId` is the SaaS `Role.id`.
- `members.value` is the app-local user ID (the `id` from the User resource).
- The custom extension carries the role's permission codes and `isSystem` flag. Apps that do not need this data can ignore it.

### Required endpoints (minimal subset)

| Resource | Endpoint                              | Purpose                                     |
| -------- | ------------------------------------- | ------------------------------------------- |
| Users    | `GET /Users?filter=userName eq "..."` | Find existing user before create            |
| Users    | `POST /Users`                         | Provision a missing user                    |
| Users    | `PUT /Users/{id}`                     | Full replace (active flag, name, email)     |
| Users    | `GET /Users/{id}`                     | Fetch user by id                            |
| Users    | `DELETE /Users/{id}`                  | Deactivate/soft-delete user                 |
| Groups   | `GET /Groups?count=500`               | List existing groups for reconciliation     |
| Groups   | `POST /Groups`                        | Create a missing role                       |
| Groups   | `PUT /Groups/{id}`                    | Full replace of membership + role extension |
| Groups   | `PATCH /Groups/{id}`                  | Partial update (membership changes)         |
| Groups   | `DELETE /Groups/{id}`                 | Remove a synced role                        |

`PUT` on a Group is the primary sync mechanism because the SaaS client always sends the full desired membership list.

### Sync semantics

- Full-state, idempotent, replace semantics.
- The SaaS worker re-reads the database for every run, so it is safe to replay, coalesce, or retry.
- Stale groups (platform-managed groups with an `externalId` that no longer has bindings) are deleted.
- Stale assignments inside a group are removed by the full-replace `PUT`.

## Mapping to Riogentix internals

- `User` create/find → `riogentix.services.database.models.user.model.User` (same logic as `resolve_tenant_users`).
- `SaasTenant` is created/updated if missing.
- `Group` → `SaasRole` (upsert by `tenant_id` + `role_id` = `externalId`).
- `Group.members` → `SaasRoleAssignment` (replace semantics for that role).
- After writing assignments, the affected users' authorization cache is invalidated so permissions take effect immediately.
